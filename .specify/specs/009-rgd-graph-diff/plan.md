## Summary

Spec 009 adds a graph diff view to the RGD Revisions tab, allowing platform
engineers to compare two `GraphRevision` snapshots side-by-side as a single
merged DAG with color-coded overlays (added/removed/modified/unchanged nodes
and edges). The tab infrastructure and backend API are already in place (spec
046, PR #275 + PR #314). What remains is: the `diffDAGGraphs` pure function,
the `RGDDiffView` merged DAG renderer, the `RevisionSelector` dropdown, diff
CSS tokens, and E2E/unit test coverage.

**Important**: The existing `RevisionsTab.tsx` already shows a side-by-side
**YAML** diff (from PR #318, GH #13 "foundation"). That is a different
deliverable from FR-005 (single merged DAG with overlays). Both features will
coexist: the YAML diff remains as a raw-diff fallback; the DAG diff is the
primary visual.

## Data Model

### Input: Two `GraphRevision` objects

Each `GraphRevision` from the API has:
- `spec.revision` (int) â€” monotonically increasing generation counter
- `spec.snapshot.spec.resources[]` â€” array of resource snapshots (same shape as RGD `spec.resources[]`)
- `spec.snapshot.spec.schema` â€” schema kind

These are fed through the existing `buildDAGGraph()` from `web/src/lib/dag.ts`
to produce two `DAGGraph` objects. `diffDAGGraphs` then takes those two graphs.

### Output: `DiffGraph`

```typescript
type DiffStatus = 'added' | 'removed' | 'modified' | 'unchanged'

interface DiffNode extends DAGNode {
  diffStatus: DiffStatus
  prevCEL?: string[]   // for modified nodes: expressions before
  nextCEL?: string[]   // for modified nodes: expressions after
}

interface DiffEdge extends DAGEdge {
  diffStatus: 'added' | 'removed' | 'unchanged'
}

interface DiffGraph {
  nodes: DiffNode[]
  edges: DiffEdge[]
}
```

### Node identity

Nodes are matched by `id` (same field kro uses as the resource identifier).
A node present in B but not A â†’ `added`. Present in A but not B â†’ `removed`.
Present in both: compare `includeWhen`, `readyWhen`, `forEach`, `template.kind`.
Any difference â†’ `modified`. All identical â†’ `unchanged`.

**No fuzzy matching** (edge case from spec: a renamed node is remove + add).
