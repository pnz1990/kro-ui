## Upstream kro node types

This spec visualizes **exactly the five node types** defined in upstream kro
(`pkg/graph/node.go`). No fork-specific concepts (e.g., `specPatch`, `stateFields`)
are represented. See design spec `000-design-system` for the visual identity of
each node type.

| Upstream `NodeType` | What it represents | CEL variable |
|--------------------|-------------------|--------------|
| `NodeTypeInstance` | The root CR generated by this RGD | `schema` |
| `NodeTypeResource` | A managed Kubernetes resource (has `template`) | `<resource id>` |
| `NodeTypeCollection` | A forEach-expanded resource set | `<resource id>` (array) |
| `NodeTypeExternal` | A referenced existing resource (`externalRef` + `name`) | `<resource id>` |
| `NodeTypeExternalCollection` | Referenced existing resources (`externalRef` + `selector`) | `<resource id>` (array) |

**Visual modifiers** (applied on top of base type, not separate types):
- **Conditional**: resource has `includeWhen` set → dashed border + `?` badge
- **ReadyWhen**: resource has `readyWhen` set → `✓` badge
- **ForEach**: resource is `NodeTypeCollection` → `∀` badge (already implied by type)

## User Scenarios & Testing

### User Story 1 — Operator views the dependency graph of an RGD (Priority: P1)

An operator navigates to `/rgds/web-service-graph` and sees an SVG dependency
graph showing the complete resource topology: root CR, managed resources,
conditional nodes, external refs, and forEach collections — each with a distinct
visual treatment derived from their upstream kro node type.

**Why this priority**: The DAG is the primary visualization of kro-ui. It is the
main reason an operator would open this tool rather than using kubectl.

**Independent Test**: Open `/rgds/web-service-graph` against a live cluster.
Confirm the root CR node appears, all child resource nodes appear with correct
labels, dependency edges are drawn between dependent pairs, and each node type
is visually distinct.

**Acceptance Scenarios**:

1. **Given** `web-service-graph` with 5 `NodeTypeResource` nodes and 1
   `NodeTypeCollection` (forEach), **When** the page loads, **Then** all 7 nodes
   are rendered in the SVG (including the root `NodeTypeInstance`) with correct
   labels and type indicators; edges connect all dependent pairs
2. **Given** a `NodeTypeResource` with `includeWhen` expressions, **When**
   rendered, **Then** the node has a dashed border and `?` badge
3. **Given** a `NodeTypeCollection` (forEach resource), **When** rendered,
   **Then** the node carries an `∀` badge and uses `--node-collection-border`
4. **Given** a `NodeTypeExternal` (externalRef by name), **When** rendered,
   **Then** the node uses `--node-external-border` (dashed) and a `⬡` icon
5. **Given** the graph is wider than the viewport, **When** rendered, **Then**
   horizontal scrolling is available and all nodes remain reachable
6. **Given** the same RGD data is rendered twice, **When** compared, **Then**
   the node positions are identical (layout is deterministic)

### User Story 2 — Operator clicks a node to inspect its CEL expressions (Priority: P1)

Clicking any DAG node opens a right-side detail panel showing the node's kind,
type badge, upstream concept explanation, and all CEL expressions on that node.
No additional API call is needed — all data is in the already-loaded RGD object.

**Why this priority**: Node inspection is the primary way an operator understands
what an RGD does.

**Independent Test**: Click a `NodeTypeResource` node. Confirm the panel shows
"Kind: ConfigMap", the "managed resource" concept explanation, and any
`readyWhen` CEL expressions highlighted in blue.

**Acceptance Scenarios**:

1. **Given** a `NodeTypeResource`, **When** clicked, **Then** the detail
   panel shows: kind badge, "managed resource" concept explanation, any
   `readyWhen` and `includeWhen` expressions highlighted with the CEL highlighter
2. **Given** a `NodeTypeCollection` (forEach), **When** clicked, **Then** the
   panel shows: kind badge, "forEach collection" concept explanation, the forEach
   dimension variable and CEL expression, and `readyWhen` uses `${each.*}` syntax
3. **Given** a `NodeTypeExternal`, **When** clicked, **Then** the panel shows:
   "external reference" concept explanation, the referenced apiVersion/kind/name
4. **Given** a `NodeTypeInstance` (root CR), **When** clicked, **Then** the
   panel shows: "root CR" concept, the schema kind/apiVersion, any `spec.schema.status`
   CEL projections
5. **Given** a node with `includeWhen: ["${foo.status.ready}"]`, **When** the
   panel opens, **Then** the `${...}` token appears in `var(--hl-cel-expression)`
   color (see spec 006)
6. **Given** the panel is open for node A, **When** node B is clicked, **Then**
   the panel updates to node B without animation lag
7. **Given** the ✕ close button is clicked, **When** the panel closes, **Then**
   the DAG takes full width again

### User Story 3 — Operator reads the raw RGD YAML with CEL highlighting (Priority: P2)

A "YAML" tab on the RGD detail page shows the full manifest with kro's custom
syntax highlighting. An operator can read it to understand the exact CEL
expressions without using kubectl.

**Why this priority**: Raw YAML inspection is the fallback for debugging anything
not surfaced by the DAG view.

**Independent Test**: Navigate to the YAML tab on `web-service-graph`. Confirm:
- `${...}` tokens are blue (`var(--hl-cel-expression)`)
- `readyWhen:` label is dark slate (`var(--hl-kro-keyword)`)
- `string` type annotations are amber (`var(--hl-schema-type)`)

**Acceptance Scenarios**:

1. **Given** a YAML with `${schema.spec.replicas}`, **When** rendered, **Then**
   the full `${...}` span is colored `var(--hl-cel-expression)`
2. **Given** a kro keyword `readyWhen:`, **When** rendered, **Then** it is
   colored `var(--hl-kro-keyword)` — visually different from regular YAML keys
3. **Given** a SimpleSchema type `string | default=primary`, **When** rendered,
   **Then** `string` → `var(--hl-schema-type)`, `|` → `var(--hl-schema-pipe)`,
   `default` → `var(--hl-schema-keyword)`, `primary` → `var(--hl-schema-value)`
4. **Given** a 400-line RGD YAML, **When** the tab renders, **Then** the YAML
   is scrollable in its container with no layout overflow and no visible
   performance degradation

---

### User Story 4 — Page has tabs for Graph, Instances, and YAML (Priority: P2)

The RGD detail page has a tab bar: "Graph" (default), "Instances", and "YAML".
The active tab is reflected in the URL query parameter so it can be bookmarked.

**Acceptance Scenarios**:

1. **Given** the page loads with no `?tab` parameter, **When** rendered, **Then**
   the "Graph" tab is active
2. **Given** `?tab=instances`, **When** the page renders, **Then** the
   "Instances" tab content is shown
3. **Given** `?tab=yaml`, **When** the page renders, **Then** the YAML tab
   content is shown
4. **Given** `?tab=<invalid>`, **When** the page renders, **Then** it falls back
   to the "Graph" tab

---

### Edge Cases

- RGD with 0 resources → show root node alone with note "No managed resources defined"
- RGD name with URL-special characters → URL-encoded in route params
- `spec.resources` field absent or null → treat as empty array; do not crash
- A resource referencing a non-existent dependency → render best-effort; show a
  warning badge on the orphaned node

## Requirements

### Functional Requirements

- **FR-001**: Page MUST fetch `GET /api/v1/rgds/:name` on mount and build the
  DAG client-side from `spec.resources` and their CEL cross-references
- **FR-002**: DAG MUST be rendered as inline SVG using a BFS-layered layout
  algorithm — no D3, no force simulation, no external graph library
- **FR-003**: The `buildDAGGraph` function MUST classify each resource into
  exactly one of the five upstream node types:

  | Classification rule | Node type |
  |---------------------|-----------|
  | Root (ID = `schema`) | `NodeTypeInstance` |
  | Has `template`, no `forEach` | `NodeTypeResource` |
  | Has `template`, has `forEach` | `NodeTypeCollection` |
  | Has `externalRef.metadata.name` | `NodeTypeExternal` |
  | Has `externalRef.metadata.selector` | `NodeTypeExternalCollection` |

- **FR-004**: The `includeWhen` modifier MUST be rendered as a visual overlay
  (dashed border + `?` badge) on top of the base node type — it is NOT a
  separate node type
- **FR-005**: Node visual tokens MUST come from `tokens.css` per design spec
  `000-design-system`. No hardcoded colors in the DAG renderer.
- **FR-006**: Clicking a node MUST open the `NodeDetailPanel` without any
  additional API call — all data is in the already-loaded RGD object
- **FR-007**: `NodeDetailPanel` MUST show: kind, type badge(s), concept
  explanation for the upstream node type, and CEL expressions highlighted via
  `KroCodeBlock` (spec 006)
- **FR-008**: A "YAML" tab MUST render the full RGD manifest via `KroCodeBlock`
  (spec 006) — no external highlighter
- **FR-009**: An "Instances" tab MUST render the instance list (spec 004 content)
  fetched from `GET /api/v1/rgds/:name/instances`
- **FR-010**: Active tab MUST be reflected in and restored from `?tab=` URL
  query parameter
- **FR-011**: The DAG layout algorithm MUST be deterministic (same input → same
  output every call)
- **FR-012**: `specPatch` MUST NOT appear anywhere — it is not an upstream kro
  concept. If encountered in a template (from a fork), render as `NodeTypeResource`
  with an "unknown template structure" badge

### Non-Functional Requirements

- **NFR-001**: DAG renders all nodes for a 10-node RGD in under 500ms after API
  response
- **NFR-002**: Node click opens detail panel in under 50ms (no I/O)
- **NFR-003**: TypeScript strict mode — `tsc --noEmit` reports 0 errors
- **NFR-004**: `DAGGraph` component MUST be purely data-driven: accepts
  `{nodes: DAGNode[], edges: DAGEdge[]}` with `nodeType` being one of the five
  upstream constants — zero kro-specific logic inside the renderer itself

### Key Components

- **`RGDDetail`** (`web/src/pages/RGDDetail.tsx`): page component, tab routing,
  fetches RGD data, builds DAG graph from `spec`
- **`DAGGraph`** (`web/src/components/DAGGraph.tsx`): pure SVG renderer,
  data-driven, no kro knowledge
- **`NodeDetailPanel`** (`web/src/components/NodeDetailPanel.tsx`): slide-in
  panel with CEL details and concept explanation per upstream node type
- **`KroCodeBlock`** (`web/src/components/KroCodeBlock.tsx`): from spec 006
- **`buildDAGGraph`** (`web/src/lib/dag.ts`): pure function converting an RGD
  `spec` object into `{nodes: DAGNode[], edges: DAGEdge[]}`. All kro-specific
  parsing lives here. Returns exactly the five upstream `NodeType` values.

## E2E User Journey

**File**: `test/e2e/journeys/003-rgd-dag.spec.ts`
**Cluster pre-conditions**: kind cluster running, kro installed, `test-app` RGD
applied. The `test-app` fixture has:
- 1 `NodeTypeInstance` root (WebApp)
- 2 `NodeTypeResource` nodes (appNamespace, appStatus)
- 1 `NodeTypeResource` with `includeWhen` → conditional (appConfig)

```
Step 1: Navigate to RGD detail (Graph tab)
  - Navigate to http://localhost:10174/rgds/test-app
  - Assert: [data-testid="dag-svg"] is visible
  - Assert: [data-testid="tab-graph"] has aria-selected="true"

Step 2: All expected nodes are rendered
  - Assert: [data-testid="dag-node-root"] is visible (NodeTypeInstance)
  - Assert: [data-testid="dag-node-appnamespace"] is visible (NodeTypeResource)
  - Assert: [data-testid="dag-node-appconfig"] is visible (NodeTypeResource, conditional)
  - Assert: [data-testid="dag-node-appstatus"] is visible (NodeTypeResource)
  - Assert: count of [data-testid^="dag-node-"] equals 4

Step 3: Conditional node has dashed border modifier
  - Assert: [data-testid="dag-node-appconfig"] has CSS class "node-conditional"
    (dashed border + ? badge)
  - Assert: it does NOT have class "node-specpatch" (that class must not exist)

Step 4: Click a NodeTypeResource node → detail panel opens
  - Click [data-testid="dag-node-appstatus"]
  - Assert: [data-testid="node-detail-panel"] is visible
  - Assert: [data-testid="node-detail-kind"] has text "ConfigMap"
  - Assert: [data-testid="node-detail-concept"] is visible
  - Assert: URL is still /rgds/test-app (no navigation)

Step 5: Close panel
  - Click [data-testid="node-detail-close"]
  - Assert: [data-testid="node-detail-panel"] is not visible

Step 6: Switch to YAML tab
  - Click [data-testid="tab-yaml"]
  - Assert: URL contains ?tab=yaml
  - Assert: [data-testid="kro-code-block"] is visible
  - Assert: at least one span.token-cel is present

Step 7: YAML tab persists on reload
  - Reload the page
  - Assert: [data-testid="tab-yaml"] has aria-selected="true"
```

**What this journey does NOT cover** (unit tests only):
- DAG layout determinism
- All five node type classifications
- CEL token color accuracy (covered by journey 006)

## Success Criteria

- **SC-001**: DAG renders all nodes within 500ms of API response
- **SC-002**: Node click opens detail panel in under 50ms
- **SC-003**: CEL expressions in YAML tab are highlighted correctly per spec 006
- **SC-004**: DAG layout is stable — 3 consecutive renders produce identical SVG
- **SC-005**: TypeScript strict mode passes with 0 errors
- **SC-006**: `buildDAGGraph` unit tests confirm `specPatch` node type is never
  produced for any input (upstream kro does not have this concept)
