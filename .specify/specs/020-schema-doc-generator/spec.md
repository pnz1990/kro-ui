# Feature Specification: RGD Schema Documentation Generator

**Feature Branch**: `020-schema-doc-generator`
**Created**: 2026-03-20
**Status**: Draft
**Depends on**: `003-rgd-detail-dag` (merged)
**Constitution ref**: §II (Cluster Adaptability), §III (Read-Only),
§V (Simplicity), §IX (Theme)

---

## Context

One of kro's key value propositions is creating custom Kubernetes APIs for
developers. But custom APIs without documentation are useless. kro's
`spec.schema` defines the full type structure: fields, types, defaults, and
status projections. This spec renders that schema as browsable API documentation
on the RGD detail page.

All data comes from the RGD's `spec.schema` field. No external API, no kro
controller access.

---

## User Scenarios & Testing

### User Story 1 — Developer reads the custom API documentation (Priority: P1)

On the RGD detail page, a "Docs" tab shows the generated API documentation for
the custom resource created by this RGD. It includes: kind, apiVersion, all
spec fields with types and defaults, and all status fields with their CEL
source expressions.

**Why this priority**: Developers consuming kro-created APIs need to know "what
fields can I set?" and "what defaults are applied?" Without this, they read raw
YAML or ask the platform team — neither scales.

**Independent Test**: With a `WebApplication` RGD whose schema defines
`spec.name: string`, `spec.image: string`, `spec.replicas: integer | default=2`,
open the Docs tab. Confirm: three spec fields are listed with their types and
the `replicas` field shows `default: 2`.

**Acceptance Scenarios**:

1. **Given** a schema with `spec.name: string`, **When** the Docs tab renders,
   **Then** a row shows: field name "name", type "string", required indicator,
   no default
2. **Given** a schema with `spec.replicas: integer | default=2`, **When**
   rendered, **Then** the row shows: field "replicas", type "integer",
   default "2", optional indicator
3. **Given** a schema with `spec.regions: "[]string"`, **When** rendered, **Then**
   the row shows: field "regions", type "[]string" (array of strings)
4. **Given** a schema with status field
   `status.ready: ${deployment.status.availableReplicas == 2}`, **When** rendered,
   **Then** the status section shows: field "ready", type "boolean" (inferred),
   source expression highlighted with CEL syntax
5. **Given** a schema with nested object types, **When** rendered, **Then**
   nested fields are shown with indentation or a collapsible tree
6. **Given** the Docs tab is open, **When** the user copies a field name,
   **Then** the field name is plain text (not wrapped in extra markup)

### User Story 2 — Developer copies example YAML (Priority: P1)

The Docs tab includes a "Example" section showing a minimal YAML instance
manifest pre-filled with the correct apiVersion, kind, and all required fields
with placeholder values. A "Copy" button copies it to clipboard.

**Acceptance Scenarios**:

1. **Given** a `WebApplication` RGD with required fields `name` and `image`,
   **When** the Example section renders, **Then** it shows:
   ```yaml
   apiVersion: v1alpha1
   kind: WebApplication
   metadata:
     name: my-webapplication
   spec:
     name: ""     # required - string
     image: ""    # required - string
   ```
2. **Given** optional fields with defaults, **When** rendered, **Then** they
   appear commented out with their default values:
   ```yaml
     # replicas: 2    # optional - integer (default: 2)
   ```
3. **Given** the "Copy" button is clicked, **When** executed, **Then** the YAML
   is copied to the clipboard without the leading/trailing code fence markers

---

### User Story 3 — Developer sees field descriptions (Priority: P2)

If the RGD schema includes OpenAPI-style descriptions (via `description` field
in the CRD generated by kro), they are shown inline with each field.

**Acceptance Scenarios**:

1. **Given** a field with a description in the generated CRD's OpenAPI schema,
   **When** rendered, **Then** the description appears below the field name in
   muted text
2. **Given** a field without a description, **When** rendered, **Then** no
   description line is shown — not a blank line, not "No description"

---

### Edge Cases

- Schema with no spec fields (only status projections) → show "This API has
  no configurable fields — all behavior is derived from status"
- Schema with deeply nested objects (3+ levels) → flatten with dot-notation
  path or collapsible tree; do NOT overflow the panel
- Schema type `string | default=` with empty default → show `default: ""`
  (explicit empty string)
- Schema uses kro SimpleSchema syntax (`string`, `integer`, `boolean`,
  `[]string`, `map[string]string`) → parse and display human-readable types

## Requirements

### Functional Requirements

- **FR-001**: Docs tab MUST read schema from the already-loaded RGD object
  (`spec.schema`) — no additional API call
- **FR-002**: Spec fields MUST be listed with: field name, type, required/optional
  indicator, default value (if present)
- **FR-003**: Status fields MUST be listed with: field name, inferred type, and
  source CEL expression highlighted via `KroCodeBlock`
- **FR-004**: Example YAML MUST be generated client-side from the schema:
  required fields with empty placeholders, optional fields commented out with
  defaults
- **FR-005**: "Copy" button MUST copy the example YAML to clipboard via the
  Clipboard API
- **FR-006**: SimpleSchema type strings MUST be parsed: `string`, `integer`,
  `boolean`, `number`, `[]<type>`, `map[<key>]<value>`, and the `| default=X`
  and `| required` modifiers
- **FR-007**: Docs tab MUST be accessible via `?tab=docs` URL param
- **FR-008**: All styles MUST use CSS tokens from `tokens.css`

### Non-Functional Requirements

- **NFR-001**: Docs tab renders within 200ms (no API call, data already loaded)
- **NFR-002**: TypeScript strict mode MUST pass

### Key Components

- **`DocsTab`** (`web/src/components/DocsTab.tsx`): schema documentation page
  with spec fields, status fields, and example YAML
- **`FieldTable`** (`web/src/components/FieldTable.tsx`): table of fields with
  type, default, required indicator
- **`ExampleYAML`** (`web/src/components/ExampleYAML.tsx`): generated example
  manifest with copy button
- **`parseSimpleSchema`** (`web/src/lib/schema.ts`): pure function that parses
  kro's SimpleSchema type strings into structured type objects
